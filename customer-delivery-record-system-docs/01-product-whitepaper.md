# 现场交付档案中心产品白皮书

## 1. 文档定位

本文不是需求愿景稿，而是基于当前源码整理的产品白皮书。目标是回答三个问题：

1. 这套系统当前到底是什么
2. 它解决了哪些真实业务问题
3. 现阶段有哪些边界、差异和后续优化方向

## 2. 产品名称与定位

### 2.1 推荐名称

`现场交付档案中心`

### 2.2 当前定位

一个面向软件公司内部交付、实施、运维、研发支持团队的 PC 端管理后台。系统围绕“客户现场项目档案”建设，把现场版本、升级、定制、配置、SQL、外部对接、脚本资产、文档附件、服务记录、问题记录统一沉淀到一个项目主档下，并通过看板和问题汇总做横向查询与统计。

### 2.3 不属于当前定位的内容

1. 不是工单流转系统
2. 不是审批流平台
3. 不是 CMDB
4. 不是通用知识库
5. 不是自动化运维发布平台

## 3. 业务背景

客户现场交付类项目通常存在以下共性问题：

1. 现场会做个性化配置、二开或临时脚本投放
2. 客户现场会与其他系统发生接口、数据库、文件或消息队列级别对接
3. 升级、补丁、SQL 和配置修改往往跨人、跨时间发生
4. 运维手册、部署说明、架构图、截图、问题过程常散落在聊天、共享盘、个人电脑里
5. 一旦换人接手，升级失败、问题复盘和交接理解成本都会明显升高

当前系统就是为了解决“记录分散、追溯困难、交接成本高、版本不清楚”这几个核心问题。

## 4. 目标用户

### 4.1 直接使用角色

1. 交付负责人：维护项目主档、关键升级和交付资料
2. 实施 / 运维工程师：录入配置、SQL、服务、问题和脚本资产
3. 研发支持：补录技术变更、问题处理、修复说明
4. 管理员：维护用户、查看登录日志、管理全局权限

### 4.2 间接受益角色

1. 售后支持
2. 新接手实施人员
3. 项目管理 / 交付管理人员

## 5. 当前系统解决的问题

### 5.1 纵向问题

围绕单个项目，系统能回答：

1. 当前现场版本是什么
2. 最近升级过几次、从哪个版本升到哪个版本
3. 保留了哪些定制
4. 配置文件改过什么
5. SQL 变更过什么
6. 放过哪些脚本、工具、补丁
7. 对接过哪些外部系统
8. 现场服务和问题怎么处理的
9. 附件、手册、部署图、截图放在哪里

### 5.2 横向问题

跨项目维度，系统能回答：

1. 本月升级次数、服务次数、问题次数是多少
2. 当前所有项目版本分布如何
3. 问题通常发生在哪些版本
4. 某个问题关键词在哪些项目里出现过
5. 某个客户或项目是否缺资料

## 8. 当前业务规则亮点

### 8.1 版本自动回写

项目当前版本不是单纯手填维护。系统会在新增、编辑、删除升级记录后，重新扫描该项目“已完成”的升级记录，并以升级时间最新的一条作为项目当前版本和最近升级时间来源。

### 8.2 变更时间自动刷新

配置、SQL、外部对接、脚本资产、服务记录、问题记录、附件上传都会刷新项目的 `last_change_at`。

### 8.3 编号自动生成

系统会自动生成统一前缀编号：

1. `PRJ` 项目
2. `UPG` 升级
3. `CFG` 配置变更
4. `SQL` SQL 变更
5. `INT` 外部对接
6. `AST` 脚本资产
7. `SRV` 服务 / 问题记录

### 8.4 问题记录与服务记录共表

当前源码把“服务记录”和“问题记录”设计成同一张业务表，通过 `service_type=incident` 区分问题记录，这让看板统计、项目概览聚合、问题汇总查询更统一。

### 8.5 备注截图不是富文本，而是结构化附件

备注框支持直接粘贴截图，但截图不会内嵌进富文本正文，而是自动落到附件中心，分类为 `screenshot`。这比富文本更利于存储、预览和审计。

## 9. 当前权限模型

### 9.1 后端真实权限

后端仍是角色驱动：

1. `admin`
2. `delivery_manager`
3. `engineer`
4. `rd_support`
5. `viewer`

并通过 JWT 中的角色信息做接口级校验。

### 9.2 当前前端表现

前端页面当前隐藏了角色编辑入口，并默认对已登录用户展示较多编辑入口；真正是否允许提交，以后端鉴权为准。

### 9.3 建议解读

这意味着“当前产品体验”偏向简化，但“当前系统实现”并没有真正移除角色模型。文档、验收和二次开发都应以“后端角色模型仍然存在”为准。

## 10. 当前实现边界与已知差异

### 10.1 角色能力与界面不完全一致

源码中后端保留角色权限，前端不再编辑角色，属于一个明显的实现差异。

### 10.2 关联升级能力预留未启用

多个表仍然保留 `related_upgrade_id` 字段，但服务层当前会强制写 `0`，前端也没有提供绑定入口。这说明源码为以后增强做了结构预留，但首版未开放。

### 10.3 新建项目与编辑项目字段不完全对称

“环境说明”当前只能在编辑项目时补录，新建时没有表单项。

### 10.4 部署环境枚举已经收敛

前端当前只使用 `standalone / cluster` 两种值，这和早期设计稿中的多种部署方式不同。当前产品文档已按现状收口。

## 11. 当前产品价值判断

如果从“内部交付档案与现场变更沉淀工具”的角度评价，这套系统已经形成了可用闭环：

1. 有项目主档
2. 有变更记录
3. 有问题追溯
4. 有资料沉淀
5. 有统计看板
6. 有审计留痕

它已经不是一个简单的 Excel 替代品，而是一个结构化的现场知识归档系统。

## 12. 后续优化建议

### 12.1 产品层建议

1. 明确“角色是否继续保留”，前后端保持一致
2. 决定是否正式启用“关联升级记录”
3. 决定是否开放更多全局汇总能力，例如附件缺失清单、脚本投放汇总

### 12.2 交互层建议

1. 增加问题汇总跳转后的定位高亮
2. 增加附件筛选与按分类视图
3. 增加全局审计查询页入口

### 12.3 研发层建议

1. 把数据库初始化 SQL 与当前前端枚举彻底统一
2. 为非管理员场景补 UI 级权限收敛
3. 为关键业务规则补更多集成测试

## 13. 结论

从当前源码看，这个系统的最佳定义不是“需求原型”，而是“已经具备实际可用性的交付知识档案系统”。文档、培训、部署和二次开发都应该围绕这一现实版本开展，而不是继续沿用早期理想化需求稿。
