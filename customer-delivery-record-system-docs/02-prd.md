## 7. 关键业务规则

1. 所有 JSON 请求使用严格解码，未知字段会直接报错。
2. 项目当前版本由最近一条“已完成升级记录”自动回写。
3. 配置、SQL、对接、脚本、服务问题、附件都会刷新项目 `last_change_at`。
4. 服务记录和问题记录共用一张表。
5. 问题版本分布不按月份过滤；问题次数、服务次数、升级次数按月份过滤。
6. 备注截图是附件，不是富文本内嵌图片。
7. 删除为逻辑删除，附件文件物理删除由服务层处理。

## 8. 交互规则

1. 列表长文本统一使用单行截断 + `title` 提示，避免单行过高。
2. 备注文本区域支持 `Ctrl+V` 粘贴截图。
3. 粘贴截图时，表单草稿值会被保留，不会清空前面已填字段。
4. 问题汇总点击“查看项目”会回到项目详情。
5. 刷新页面时，登录页路径为 `/login`，避免首页闪回登录页。

## 9. 已知实现差异与优化建议

1. 用户角色在后端存在，但前端未暴露角色编辑，应尽快统一策略。
2. `related_upgrade_id` 已有字段但未开放使用，建议后续决定保留还是正式启用。
3. 新建项目与编辑项目字段不完全一致，后续可统一。
4. 数据字典表存在于历史 SQL 中，但当前运行期未真正消费，若长期保留建议补服务层接入。
